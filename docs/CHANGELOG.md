## 2026-06-09  Stream C slice 3: generateCurrentWeek bridges engine to Convex

`app/convex/generateWeek.ts` ships `generateCurrentWeek`, an `internalMutation` that calls the engine's `generateWeek` and persists the result into the `currentWeek` table. Inputs: `weekStart` (ISO Monday), optional `rng` (a numeric value wrapped as a constant-returning closure for deterministic Saturday alternation in tests), and optional `userRequestedDishId` (for ┬¦3.2 substitution). The function reads the baked library from `@plantry/engine/library` and the seed history from `@plantry/engine/history`, derives the Bangalore season from the ISO date string directly (no Date object), calls the engine, then converts: engine days are already short-form so the day map is identity; meal lowercase ("Breakfast" -> "breakfast"); per slot it emits `{ day, meal, dishId: slot.dishes[0].id, customLabel: null, source: "generated", author: "system", updatedAt }`. Slots whose `dishes[]` ended up empty (cap dropped them) are omitted. Replacement semantics: if a row exists for the same `weekStart`, it is deleted before insert; new row starts at `version: 1`, `status: "draft"`. Engine warnings (e.g. "Friday over cap (5), dropped: Bread omelette") become one `incidents` row each with `source: "engine"`, `severity: "warn"`, `context: { weekStart, weekId }`. Return shape: `{ weekId, version, incidentCount }`. Not browser-callable; the EM (or a future Stream F scheduler) invokes via `npx convex run generateWeek:generateCurrentWeek '{"weekStart":"YYYY-MM-DD"}'`. (#20)

## 2026-06-09  Stream B slice 6: item cap per docs/engine.md ┬¦5

`engine/src/cap.ts` exports `applyCap({ slotsByDay }): { slotsByDay; droppedDishIds[] }` plus `WEEKDAY_CAP = 5` and `SATURDAY_CAP = 3`. When a day exceeds its cap, dishes are dropped one at a time: lowest Satiety first; among those the longest Prep Min; final tie-break (silent in ┬¦5 itself) drops the later array position so earlier slots' picks win. The function is pure and returns a fresh `Map<Day, Dish[]>` plus the dropped ids. Sunday is passed through unchanged (defensive, since ┬¦2 schedule emits no Sunday slots). `SlotPick` is exposed as a `Dish` alias so the in-flight `generateWeek` slice can widen it to carry per-slot metadata without rewriting the cap algorithm. 14 unit tests cover under-cap passthrough, at-cap passthrough, weekday + Saturday over-cap, tie-break ladder, multi-day independence, and the "set difference" permutation property. Engine suite now 140 tests across 12 files. (#18)

## 2026-06-09  Stream B slice 5: ingredient consolidation per docs/engine.md ┬¦6

`engine/src/consolidation.ts` models the weekly ingredient buy-list state and provides preference scoring against it. `IngredientLedger` is a `Map` keyed by canonical ingredient name from the pack-size header; entries carry `packSizeGrams`, `packsOnBuyList`, `usedGrams`, `leftoverGrams`. `emptyLedger(packSizes)` initialises one tracked entry per pack-size row; untracked ingredients (everything not in the header: fresh produce, pantry staples) never enter the ledger per ┬¦6 wording. `applyPick(ledger, dish, ingredients)` walks the dish's ingredient rows in grams, adds enough packs to cover usage (rounded up to the next pack multiple), and recomputes leftover. `scoreCandidates(pool, ledger, ingredients)` orders candidates by how many ledger entries above the 50 g threshold they would consume; ties preserve input order. `scoreSoftConsolidation(pool, lastFreshItemsUsed)` adds the ┬¦6 soft preference on capsicum, tomato, cucumber, onion, mint, coriander; canonical mint/coriander names in `data/ingredients.md` are "Mint Leaf" and "Coriander Leaf" and both forms are in the constant. `rankByConsolidation` composes hard score primary, soft score secondary. The ┬¦4 step 3 placeholder shipped in B slice 4 is now wired to `byIngredientConsolidation` when a ledger is supplied; absent ledger preserves slice-4 no-op behaviour. Unit handling assumption documented inline: only `unit === "g"` ingredient rows are consolidated against gram pack sizes; ml / pcs rows fall back to no consolidation (no tracked ingredient is ml or pcs today; the guard is defensive). 23 new consolidation tests plus 4 priority extension tests; engine suite now 126 tests across 11 files. (#16)
