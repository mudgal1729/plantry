## 1. What Plantry is today (phase 1)

Plantry is a weekly meal planner for a two-adult Indian household in Bangalore. Once a week it generates a Monday to Saturday menu (breakfast and lunch) from a fixed dish library, following a set of composition and selection rules, then produces a shareable menu image and a grocery list.

Today it runs inside Claude Cowork as a chat workflow over a folder of markdown files:

- `instructions.md`: folder-level operating instructions (the weekly loop, image format, grocery-list format).
- `rules.md`: the full generation logic (eligibility, weekly schedule, slot composition, selection priority, item cap, ingredient consolidation, field reference). This is the heart of the system.
- `dishes.md`: the dish library. One row per dish; about 90 active rows. Columns: `ID, Name, Category, Time, Tags, Primary Ingredient, Preferred, Active, Satiety, Prep Min, Seasons`.
- `ingredients.md`: per-dish ingredient quantities (`Dish ID, Dish Name, Ingredient, Quantity, Unit`), plus a header table of tracked ingredients and their pack sizes (Paneer 200 g, Mushroom 200 g, Lettuce 100 g, Curd 500 g, Chicken Keema 500 g, Chicken Breast 250 g, Fish 500 g, Prawn 500 g).
- `menu_history.md`: append-only record of past weeks; one row per dish per day (`Week Start, Day, Meal, Dish Name, Dish ID`). Drives the no-repeat (recency) logic.
- `learnings.md`: design notes and the long-term PWA direction. Read section "Long-term direction (PWA)"; this handoff builds directly on it.

How a week runs today: Claude reads the four data files, applies `rules.md` to produce the menu, Rajat tweaks it conversationally, and on finalise Claude writes a PNG, prints the grocery list, and appends the week to `menu_history.md`.

The phase-1 system works and has run for several weeks. Phase 2 is about making it usable by Tuhina too, outside Cowork, with both people able to see and adjust the plan.

## 2. Phase 2 goal

Make Plantry a small two-user web app:

1. Both Rajat and Tuhina can open a website (on phones) and see the current week's menu.
2. Either can adjust this week's plan within bounded controls (swap a dish, drop in a one-off, leave a comment).
3. Edits to *this week* never touch the rules or the library directly. Operational changes are recorded against the week. Structural feedback (comments implying a library or rule change) is recorded as a queued request, not applied.
4. A separate, deliberate review step reads the queued requests and proposes how the underlying structure (`dishes.md`, `ingredients.md`, `rules.md`, and a new `changelog.md`) should change, as a diff Rajat approves before it lands.

The guiding principle, already stated in `learnings.md`: disruptions and preferences are handled as *inputs to the engine and a review queue*, not by piling edge cases into `rules.md`. Keep the rules clean; push variability into data and user controls.

## 3. Decisions locked with Rajat (build to these)

These four were decided up front. Do not relitigate them without checking in.

**3.1 Generation is ported to a deterministic code engine.**
Rewrite the logic in `rules.md` as real program code so that generation is deterministic (identical inputs always yield identical output), reproducible, instant, and free to run (no model call needed to produce a weekly menu). `rules.md` stays as the human-readable specification; the engine is the executable implementation of it. The two must stay in sync: any change to one is a change to both, and the review loop (section 8) is responsible for keeping them aligned.

**3.2 The data store stays markdown in git.**
Keep `dishes.md`, `ingredients.md`, `menu_history.md`, etc. as the canonical source of truth, living in a git repository (move the folder into a GitHub repo as step zero). Git gives a full audit trail (who changed what, when) and easy rollback for free, which suits a two-person household. The app parses these files into memory at runtime and writes changes back as commits. The known weakness is concurrent writes (two people editing the same file at the same instant); section 7.4 covers how to handle that given the on-demand, two-person usage pattern.

**3.3 The review loop is on-demand and produces a diff to approve.**
Recorded comments and change requests are not auto-applied. When Rajat chooses to run a review, the system reads the queued requests and proposes a before/after diff across the affected files for Rajat to approve before anything lands. The natural implementation is a GitHub pull request (see section 8): the proposal *is* a PR, and approval *is* a merge.

**3.4 Dashboard v1 feature set.**
A shared read-only current-week view (assumed in all cases), plus three interactive features:

- Dish swap with auto-suggest (engine offers valid alternatives for that slot).
- Custom one-off entry (type a dish for a single day without adding it to the library).
- Comments that feed the review loop.

Explicitly out of v1 scope, planned as a fast-follow: day-level overrides (skip a day, mark eating out, swap two days). `learnings.md` rates these highly (they solve roughly 80 percent of disruptions), so design the data model and UI so they slot in cleanly later, but do not build them in v1.

## 4. Target architecture

Think of Plantry as two loops over one git-backed dataset.

**The fast loop (operational, immediate).** What is happening this week. Viewing the menu, swapping a dish, adding a one-off, leaving a comment. These are quick, low-stakes, and apply right away (an operational change is committed to the week's record; a comment is queued). The engine and the live menu state serve this loop.

**The slow loop (structural, deliberate).** How the system itself should evolve. Comments accumulate ("we never actually cook lauki", "this was too spicy", "we should add a low-spice option for sick days"). On demand, the review step turns that accumulated signal into proposed edits to the library and rules, as an approvable diff. This loop is where `dishes.md`, `ingredients.md`, `rules.md`, and `changelog.md` change.

Keeping these loops separate is the whole point of Rajat's "record, do not apply directly" instruction. The fast loop must never silently mutate the rules; the slow loop is the only path by which structure changes, and it always passes through human approval.

### 4.1 Components

1. **Web frontend (installable PWA).** A Progressive Web App: a normal website built so it can be added to a phone home screen and run full-screen like a native app, with the current week cached so it opens instantly even on poor network. This avoids any app-store process; same web code serves both phones. Renders the current-week view and the three v1 controls.

2. **Backend / API.** A small server that: parses the markdown data files into structured objects, serves the current week and the dish library to the frontend, accepts operational edits and comments, runs the engine to generate a week or to produce swap suggestions, and writes changes back to the git repo as commits with author attribution.

3. **Generation engine (deterministic, shared library).** A standalone module implementing `rules.md`. Pure functions: given the library, history, season, and any locked or overridden slots, it returns a complete valid week, or for a single slot it returns the ranked candidate set used by auto-suggest. No I/O, no model calls, fully unit-testable. Used by the backend for both full generation and dish-swap suggestions.

4. **Data layer (git markdown).** The repo of canonical files. The backend reads at startup and on demand, and commits writes. Treat the markdown tables as a serialisation format: write parsers and serialisers so a round trip (parse then re-serialise) is lossless and produces minimal diffs.

5. **Review loop (on-demand, human-approved).** Reads the queued change requests and proposes structural edits as a diff / pull request. This is the slow loop. It can be run by Claude Code with the GitHub connector (section 8).

### 4.2 Data flow (text diagram)

```

                     |
              (PWA frontend)
                     |
        view week / swap / one-off / comment
                     |
                 Backend API
        /         |              \

        |         |                  |
        v         v                  v
```

```
    src/...
    test/...             # rules encoded as tests (see section 6)
  app/
    server/              # NEW: backend API, markdown parse/serialise, git writes

  HANDOFF_phase2.md      # this file
```

A note on Vite: the frontend build tool already noted in Rajat's knowledge base. Use it for the web app dev server and bundling.

### 5.1 `changelog.md` (new)

The audit of how the *structure* evolved over time, distinct from `menu_history.md` (which records operational weeks). One entry per approved structural change: date, what changed (dishes added/retired, ingredient pack-size update, rule edit), the triggering comment(s), and a one-line rationale. This is the human-readable story of the library and rules evolving. The review loop appends to it whenever it lands a change.

### 5.2 `change_requests/` (new)

The inbox for the slow loop. Each queued comment or structural suggestion becomes a small file (or a row in an append-only log; pick one and be consistent) capturing: timestamp, author (Rajat or Tuhina), the dish/day it was attached to (if any), the free-text comment, and a status (`queued`, `in_review`, `applied`, `dismissed`). Operational edits do not go here; only feedback that might change structure. The review loop consumes `queued` items.

### 5.3 `data/weeks/<week-start>.md` (new)

The live state for a given week: the generated menu plus any operational overrides (swaps, one-offs) and a status (`draft`, `final`). Separating this from `menu_history.md` lets the week be edited freely while in `draft`; on finalise, its dishes are appended to `menu_history.md` (preserving the existing history format and the recency logic that depends on it). Stamp each override with author and timestamp, as `learnings.md` anticipates ("History as audit log: once collaborative, history needs edited_by and timestamps").

## 6. Porting the engine (rules.md to code)

`rules.md` is already written almost like a spec; encode each numbered section as engine logic and, crucially, as tests. Suggested mapping:

- **Section 1, eligibility.** Filter dishes by `Active = Yes` and season membership. Implement the Bangalore season calendar (Summer Mar to May, Monsoon Jun to Sep, Winter Oct to Feb; `All` means year-round). Current date is the input that picks the season.
- **Section 2, weekly schedule.** The fixed grid (Mon/Wed/Fri: 2 breakfast + 3 lunch; Tue/Thu: 1 breakfast + 4 lunch; Sat: 3 lunch; Sun: none). The Saturday Menu 3 / Menu 4 alternation reads the most recent Saturday from history and picks the other. The "at most one weekday lunch may substitute Menu 3/4" rule (section 2 and 3.2) is a constrained option, not a default.
- **Section 3, slot composition.** Breakfast Options A/B/C and the four lunch menus are candidate-set builders: each returns the set of dishes eligible for that slot given the tags/categories it requires (`HP`, `complete_meal`, `complete_carb`, `fruit`; Categories like Gravy dish, Dry dish, Keto, Accompaniment, Dessert, Chapati, Rice). These internal labels never appear in output (`learnings.md`: "decouple display from structure").
- **Section 3.1, lunch carb rule.** Default Chapati; Rice at most once per week; carbs are recency-exempt.
- **Section 4, selection priority.** The ranking pipeline: longest-unused first (from history), then same-day primary-ingredient deprioritisation, then ingredient consolidation, then `Preferred = Yes` as a tiebreaker. `fruit` and lunch carbs are recency-exempt.
- **Section 5, item cap.** 5 per weekday, 3 on Saturday; if over, drop lowest-Satiety then longest-Prep-Min until at cap.
- **Section 6, ingredient consolidation.** Tracked ingredients are exactly those in the `ingredients.md` pack-size header; 50 g leftover threshold; prefer the next slot that consumes leftover; soft preference for shared fresh produce. This is a soft, low-priority rule (see `learnings.md`: "soft rules need soft language"); keep it as a tiebreaker, not a hard constraint.

**Auto-suggest reuses this engine.** A dish swap is just "rebuild the candidate set for this one slot, exclude the current pick and anything in the no-repeat window, return the ranked alternatives." Do not write a second code path for suggestions; expose the engine's per-slot ranking and let the UI present the top few.

**Validation parity.** Before trusting the port, regenerate the historical weeks in `menu_history.md` from their inputs and confirm the engine produces valid menus under the same rules (exact dish-for-dish match is not required because phase 1 had Claude's judgment in the loop; rule-validity is). `learnings.md` records that a 5-week forward simulation previously surfaced four subtle bugs (rice + dry-only conflict, keto pool too small, mislabeled HP carbs, dry HP leaving no gravy); reproduce that simulation as a test harness so regressions are caught.

## 7. Feature spec (v1)

### 7.1 Shared read-only current-week view

Both users see the same current week: six day cards (Mon to Sat), each showing breakfast and lunch dishes, with a date badge, plus the grocery list below. No internal labels (no "Menu 3", no "weekend", no ingredient-reuse callouts), matching the phase-1 image format in `instructions.md` and `learnings.md`. This is the foundation; build it first. The grocery list is generated from `ingredients.md` aggregated across the week's dishes, grouped in the fixed order (Proteins and Dairy, Pantry, Vegetables, Aromatics and Herbs, Other), with tracked ingredients rounded up to the next pack multiple and common staples omitted, exactly as `instructions.md` specifies.

### 7.2 Dish swap with auto-suggest

On any dish in the current week, the user can request a swap. The engine returns valid alternatives for that slot (correct category/tags, in season, not in the no-repeat window), ranked by the section 4 priority. The user picks one; the swap is an operational override recorded against the week (`data/weeks/<week>.md`), stamped with author and time. It does not modify the library. Re-run the grocery list after a swap.

### 7.3 Custom one-off entry

The user can type a free-text dish for a single day (leftovers, ordered-in, a guest recipe). It occupies the slot for that week only, is clearly marked as custom, and is never added to `dishes.md`. It carries no ingredient data, so either exclude it from the grocery list or let the user add a free-text shopping note. If a custom entry recurs and should become a real dish, that is a structural change and goes through the review loop (the user can attach a comment to that effect).

### 7.4 Comments that feed review

On any dish or day, the user can leave a comment. Comments do nothing immediately; they are written to `change_requests/` as `queued`, with author, timestamp, and the attached dish/day. Comments are the raw material of the slow loop. Surface them lightly in the UI (for example a small marker on a commented dish) so both users can see what feedback exists, but keep them out of the menu output itself.

### 7.5 Two-user concerns

- **Identity and attribution.** Keep it light for a two-person household. A simple profile pick (I am Rajat / I am Tuhina) stored on the device is enough to attribute edits and comments; a shared passcode on the site is enough to keep the public URL private. No per-user accounts or heavy auth in v1. `learnings.md` explicitly wants attribution on edits and history.
- **Concurrency.** With on-demand, two-person usage, simultaneous edits to the same week are rare but possible. Use optimistic concurrency: when the app loads the week it records a version (for example the latest git commit hash or a monotonically increasing revision in the week file); on save it checks the version is unchanged and, if not, refuses and reloads rather than overwriting the other person's edit. This is simpler and safer than locking and is sufficient at this scale. Do not attempt automatic three-way merges of menu edits.

## 8. The review loop (slow loop) spec

This is the mechanism for section 3.3 and Rajat's core "record, then review" requirement.

**Trigger.** On demand, initiated by Rajat (a button in the app, or simply a Claude Code session). Day-to-day use never triggers it.

**Input.** All `queued` items in `change_requests/`, plus the current state of `dishes.md`, `ingredients.md`, `rules.md`, `menu_history.md`, and recent week files for context.

**What the review does.** Read the accumulated comments, cluster them into themes, and decide what structural change (if any) each implies. Examples of the reasoning expected:

- "We never actually cook lauki" appearing repeatedly implies setting those dishes `Active = No`, not deleting them.
- "Too spicy on sick days" implies a possible new `low_spice` tag plus a small rule addition, which `learnings.md` flags as the right pattern (new constraints become tags plus a small rule, never architectural changes). Note in the proposal that adding a tag means touching both `rules.md` and the engine code.
- A recurring custom entry implies adding a real dish row to `dishes.md` and its quantities to `ingredients.md`.
- A pack-size correction implies editing the `ingredients.md` header.

Apply the discipline already in `learnings.md`: prefer fixing one or two rows at the data level over adding a column or a cross-cutting rule; do not generalise from a single edge case; do not materialise data that can be inferred.

**Output.** A proposed diff across the affected files (`dishes.md`, `ingredients.md`, `rules.md`, the engine code if a rule changed, and an appended `changelog.md` entry), presented for approval. The clean implementation: open a GitHub pull request. A pull request is exactly the change-request pattern Rajat described; it shows the before/after of every file, Rajat reviews it, and merging it is the approval. Claude Code already has a GitHub connector, so the review session can read the inbox and open the PR directly. On merge, mark the consumed `change_requests/` items `applied` and (if the rules changed) make sure the engine tests are updated in the same PR so spec and code stay in lockstep.

**Why it stays human-approved.** Structural changes alter every future menu, so they must never auto-apply. The fast loop is reversible and local to one week; the slow loop is not, which is why it always ends at a human-reviewed diff.

## 9. Suggested build phases

1. **Repo and parsers.** Move the folder into a GitHub repo. Write loss-minimal parsers and serialisers for `dishes.md` and `ingredients.md` (and the history format). Round-trip tests first.
2. **Engine.** Port `rules.md` to the deterministic module with the section-by-section tests and the multi-week simulation harness from section 6.
3. **Read-only PWA + backend.** Serve and render the current week and grocery list (section 7.1). Installable on phones. This alone delivers the highest-value item in `learnings.md` (the shared read-only view).
4. **Operational edits.** Dish swap with auto-suggest (7.2) and custom one-off entry (7.3), with optimistic-concurrency saves and author attribution.
5. **Comments and the review loop.** Comment capture into `change_requests/` (7.4), then the on-demand review producing a PR (section 8), with `changelog.md`.
6. **Fast-follow (post v1).** Day-level overrides (skip / eat out / swap days); calendar awareness later, as `learnings.md` outlines.

Ship phases 1 to 3 as the first usable milestone (Tuhina can see the week on her phone), then layer 4 and 5.

## 10. Open questions for the engineer to resolve or raise

These were intentionally left to the rebuild. Flag a recommendation on each.

1. **Hosting.** Rajat wants phone access from anywhere, so a small always-on host is implied (a managed platform, or a home machine exposed safely). Recommend the simplest option that keeps the git repo as the writable store and stays near-free at this scale; note the privacy and cost tradeoff.
2. **Backend shape.** A single small persistent server (simplest given git commits and a long-lived working copy of the repo) versus serverless functions (which complicate holding a git working tree). Recommend one.
3. **Markdown as write target.** Confirm that committing parsed-then-reserialised markdown yields clean, reviewable diffs. If round-trips prove noisy, consider keeping the human-edited files canonical and writing machine state (week files, change requests) as a tidier format, while never auto-rewriting `rules.md`.
4. **Grocery list for custom entries.** Decide whether custom one-offs contribute a free-text shopping note or are simply excluded (section 7.3).
5. **Review trigger.** In-app button that kicks off a Claude Code review, versus Rajat running the review session manually. Either is fine for v1; recommend based on effort.
6. **Engine and rules.md sync enforcement.** Propose a lightweight check (for example a test that fails if `rules.md` is edited without a corresponding engine change in the same PR) so the spec and the code cannot silently drift.

## 11. Glossary

A few terms used above, explained for completeness.

- **PWA (Progressive Web App).** A website built so a phone can "install" it to the home screen and run it full-screen like a native app, with the page cached for instant, offline-tolerant loads. Avoids the app stores entirely; one web codebase serves both phones.
- **Deterministic engine.** Program logic where identical inputs always produce identical output, with no randomness and no model call. The opposite of having Claude reason freshly each time. Buys reproducibility, speed, zero per-run cost, and testability, at the cost of expressing fuzzy judgment in code.
- **Git as a data store.** Using version-controlled text files as the source of truth instead of a database. Every change is recorded with author and timestamp, giving a free audit trail and easy rollback; the tradeoff is weak handling of simultaneous writes and of ad-hoc queries.
- **Optimistic concurrency.** A way to handle two people editing at once without locking: each editor remembers the version they loaded, and a save is rejected if the underlying version has changed since, prompting a reload rather than silently overwriting the other person's work.
- **Change-request / diff-approval pattern (pull request).** Changes are not written straight into the source of truth; they are proposed as a reviewable before/after (a diff) that a human approves before it lands. A GitHub pull request is the standard implementation: propose, review the diff, merge to apply.
- **Monorepo.** A single repository holding multiple parts of a project (here: the data files, the engine, and the app), so they share one history and version together.
